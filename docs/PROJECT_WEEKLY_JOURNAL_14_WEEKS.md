# Healthcare Agent 毕业设计周记

## 第1周：明确选题方向与系统目标

本周主要围绕毕业设计题目进行需求梳理和可行性分析。最初确定的方向是“面向健康评估的智能 Agent 系统”，核心目标不是简单做一个聊天机器人，而是构建一个能够结合用户健康信息、医学知识库和多阶段推理流程的健康评估工具。通过查阅现有大语言模型在医疗问答、健康管理和检索增强生成方面的应用，我认识到直接调用通用大模型虽然实现简单，但在医学依据、风险提示和回答稳定性方面存在不足，因此系统设计需要加入知识库检索、专科分流和安全边界控制。

在技术方案上，本周初步确定使用 Python 作为后端开发语言，使用 LangChain 调用 DeepSeek 模型完成 Agent 推理，使用 FastAPI 暴露服务接口，并预留前端交互页面。知识库方面计划采用本地 JSON 文件保存医学知识条目，再通过向量检索方式为模型提供依据。经过分析，系统需要覆盖睡眠、运动、尼古丁接触、饮食、BMI、血压、血脂、血糖、心理健康和社会决定因素等健康主题。

本周的主要收获是明确了系统边界：本项目用于健康评估和风险提示，不替代医生诊断；回答必须尽量基于用户提供的信息和知识库依据，尤其在高风险场景下要明确提示及时就医。下一步计划开始搭建项目目录、环境配置和基础命令行入口，为后续模块化开发打好基础。

## 第2周：搭建项目基础框架与运行环境

本周完成了项目的基础工程搭建。首先创建了 `src/healthcare_agent/` 作为核心代码目录，并将配置、Agent 编排、知识库、数据预处理、API、数据库、认证、会话服务等功能预先拆分为独立模块。项目使用 `pyproject.toml` 管理依赖，明确 Python 版本要求为 3.12 及以上，并引入 LangChain、DeepSeek、FastAPI、Chroma、sentence-transformers、pypdf、pytesseract、PyMySQL 等依赖。

配置部分实现了 `.env` 环境变量读取，方便管理 DeepSeek API Key、MySQL 连接信息、RAG 开关、知识库路径、Chroma 持久化目录、Embedding 模型和 Reranker 模型等参数。为了方便开发和测试，项目保留了 `main.py` 作为命令行入口，`run_api.py` 作为 FastAPI 启动入口，同时编写 README 记录安装、启动和常用命令。

本周还初步规划了系统响应结构，准备使用 Pydantic schema 统一请求、响应、Token 统计、知识片段和专科评估结果等数据格式。这样做可以减少后续前后端对接时的字段混乱，也方便评测脚本读取系统输出。

本周遇到的问题主要是模块边界如何划分。经过调整，决定把“Agent 推理流程”和“HTTP API”分离，把“知识检索”和“Prompt 构造”分离，使后续修改某一部分时不会影响整体结构。下一周将重点收集和整理医学知识数据，并开始构建本地知识库。

## 第3周：收集健康评估知识并整理知识库结构

本周进入医学知识库建设阶段。根据系统目标，我围绕生命 8 要素和健康评估常见问题收集整理了多类知识数据，包括血压、血糖、血脂、健康饮食、BMI、减重、体育锻炼、睡眠、戒烟指南、高血压治疗方案、糖尿病治疗方案、高血脂治疗方案，以及心理健康与社会决定因素相关内容。这些资料被整理为 `knowledge_base/` 目录下的 JSON 文件。

在知识库格式设计上，我没有简单把资料保存为纯文本，而是采用带有层级结构的 JSON。每个文件包含 `topic`、`children`、`content` 等字段，并增加 `agent_tags` 标记该知识适合被哪些专科 Agent 使用。例如血压、血糖、血脂主要标记为 `cardiometabolic_health`，健康饮食和 BMI 标记为 `diet_bmi`，睡眠、运动和戒烟相关内容标记为 `sleep_activity_nicotine`，心理与社会因素标记为 `mental_social_health`。

本周也对知识库内容进行了清洗和归类，尽量让每个条目表达清楚、数值标准明确、主题边界稳定。健康评估系统对知识依据要求较高，如果知识库条目太散或主题标记不准确，后续 RAG 检索会受到明显影响。因此本周重点不是写代码，而是保证知识数据能支撑后续推理。

本周完成后，系统已经具备了本地医学知识库的雏形。下一步计划实现 JSON 知识库加载、层级内容扁平化、文本切分和向量索引构建。

## 第4周：实现知识库预处理与 RAG 索引构建

本周主要实现知识库处理模块 `knowledge_base.py`。该模块负责扫描 `knowledge_base/` 目录下所有 JSON 文件，将层级 JSON 转换为可检索的文本片段，并写入 Chroma 向量数据库。为了适配不同复杂度的 JSON 文件，我实现了递归解析逻辑：对于较短的结构，直接扁平化为一个知识片段；对于较长的结构，继续按子节点拆分；对于超长文本，再按固定长度和重叠窗口切分，避免单个片段过长影响检索。

索引构建时，系统使用 `BAAI/bge-small-zh-v1.5` 生成向量表示，并将 `source_file`、`section_path`、`agent_name` 等元数据一起写入 Chroma。这样后续检索结果不仅能提供知识内容，还能告诉用户依据来自哪个知识文件和哪个层级路径。为了支持专科过滤，构建索引时会根据 `agent_tags` 为同一知识片段写入对应专科标签。

本周还实现了知识库状态查询和自动构建机制。当索引不存在或为空时，系统可以根据配置自动构建索引；同时提供 `python main.py --rebuild-kb` 和 `python main.py --kb-status` 等命令，方便开发阶段检查知识库是否可用。

本周遇到的主要问题是如何平衡知识片段粒度。片段太大时会影响检索精度，片段太小时又容易丢失上下文。最终采用约 700 字符、120 字符重叠的切分方式，并保留 JSON 层级路径辅助理解。下一周将开始实现健康数据采集与文件解析模块。

## 第5周：设计健康数据输入与文件解析模块

本周围绕“健康数据采集模块和数据解析模块”展开开发。系统需要支持用户直接输入文字，也需要支持上传病例、体检报告或截图等文件。因此我实现了 `preprocessing.py`，统一处理文本、CSV、PDF 和常见图片格式，最终将不同来源的数据合并为一段标准化的医疗输入。

对于 `.txt` 文件，系统支持 UTF-8、UTF-8-SIG、GBK、GB18030 等常见编码，避免中文文件读取乱码。对于 `.csv` 文件，系统使用 CSV 解析器读取每一行并清理空字段，将表格内容转换为便于模型理解的文本。对于 PDF 文件，系统使用 `pypdf` 提取各页文本，并保留页码信息。对于图片文件，系统调用本机 Tesseract OCR，并使用 PIL 做灰度化、自动对比度、锐化、放大和阈值化等预处理，以提高体检报告截图的文字识别质量。

本周还加入了文件类型校验、空文件校验、无法提取文本时的错误提示，以及 OCR 语言包检测。由于 OCR 结果可能存在错字、错行和重复，系统不会直接把 OCR 结果当作最终医学判断，而是把它作为原始资料交给后续 rewrite agent 进行整理。

通过本周开发，系统输入方式从单一文本扩展为多模态健康资料输入，为后续前端上传病例文件和综合健康评估打下基础。下一周计划继续完善标准化输入流程，并设计统一的 API 响应字段。

## 第6周：完成多源健康数据标准化与接口契约设计

本周在上一周文件解析的基础上，继续完善健康数据标准化流程。系统通过 `build_preprocessed_input` 将用户直接输入的文字和上传文件提取内容合并，记录 `source_summary`、`preprocessed_text` 和 `preprocessing_notes`。这样后续无论用户输入来自文本、PDF、CSV 还是图片，Agent 都可以拿到统一格式的数据。

为了让前端和评测脚本能够稳定使用系统输出，本周重点完善了 `schemas.py` 中的 Pydantic 数据结构。系统响应不仅包含最终回答 `content`，还包含 `agent_name`、`routed_agent_names`、`route_reason`、`rewritten_query`、Token 消耗、推理耗时、知识库片段、粗召回片段、重排片段和专科分析结果等字段。这些字段后续在实验评估中非常重要，例如可以直接统计路由是否正确、检索到了哪些知识文件、模型输出耗费多少 Token。

本周也初步设计了 API 契约，区分专科路由流程和全科综合流程，并为普通 JSON 调用和 SSE 流式调用预留接口。文件上传接口统一使用 multipart form-data，既可以传文本，也可以传多个文件。

本周的关键收获是认识到“标准化处理”不仅是清洗文本，也包括稳定数据结构和系统边界。只有输入输出格式固定，后续 Agent、前端、数据库和评测模块才能顺利协作。下一周将正式搭建基于大语言模型的 Agent 核心架构。

## 第7周：构建 Agent 核心架构与 Prompt 体系

本周开始实现系统最核心的 Agent 流程。根据前期设计，系统采用“问题改写、专科路由、专科分析、综合总结”的多阶段架构，而不是把用户输入直接发送给模型。首先实现 `rewrite_medical_data`，用于将口语化描述、PDF 提取文本或 OCR 文本整理成清晰的医疗输入。该步骤要求保留原始事实，不添加新病史、不做诊断，只负责去噪和结构化。

随后实现 `route_to_specialists`，由路由 Agent 根据整理后的输入选择需要参与分析的专科方向。系统目前包含四个专科 Agent：睡眠/活动/尼古丁、饮食/BMI、血压/血脂/血糖、心理健康/社会决定因素。路由 Prompt 要求模型输出严格 JSON，并限制最少 1 个、最多 4 个专科，避免无意义全选。

本周还编写了各专科 Prompt 和全科 Prompt。专科 Prompt 要求每个 Agent 只重点分析用户实际提供的信息，避免大篇幅讨论缺失数据；全科 Prompt 则从生命 8 要素、心理健康和社会因素进行整体评估。为了控制医疗风险，我加入 RAG guardrail，要求治疗方案、指标阈值、风险分层等关键判断必须依据知识库检索结果，依据不足时要明确说明。

本周完成了 Agent 核心架构的第一版，为后续多专科并行、知识检索和总结 Agent 集成奠定基础。下一周将完善并行执行、Token 统计、流式事件和中期检查材料。

## 第8周：完善多 Agent 编排并准备中期检查

本周继续完善 `agent.py` 中的多 Agent 编排。专科流程的完整执行顺序被固定为：先进行输入改写，再通过 router 选择专科，然后并行调用相关专科 Agent，最后由 summary Agent 汇总为最终答复。为了提高效率，专科分析使用 `asyncio.gather` 并行执行，适合同时涉及多个健康主题的复杂输入。

在知识库接入方面，每个专科 Agent 会先按自身 `agent_name` 检索对应知识片段，再把重排后的知识上下文注入 Prompt。系统同时保留粗召回结果和 reranker 重排结果，方便后续调试和实验分析。总结 Agent 会接收用户整理后的健康信息、各专科分析结果和合并后的知识片段，输出更清晰的综合建议。

本周还实现了 Token 使用量聚合、推理耗时统计、路由结果排序、知识片段去重和 fallback 路由机制。当模型路由输出不是标准 JSON 时，系统会尝试从文本中解析专科名；如果仍失败，则根据关键词进行兜底路由，避免系统直接中断。

为准备中期检查，本周同步整理了 `docs/DEVELOPMENT_ALIGNMENT.md`，说明系统边界、运行环境、Agent 流程、API 契约、知识库格式和前端解析注意事项。中期阶段系统已经具备命令行评估、RAG 检索、多专科分析和综合总结能力，后续重点转向后端接口、数据库持久化和网页交互。

## 第9周：集成 FastAPI 服务与知识库管理接口

本周重点将 Agent 能力封装为可调用的后端服务。通过 `api.py` 创建 FastAPI 应用，提供专科评估和全科评估接口，包括普通 JSON 返回和 SSE 流式返回两种形式。普通接口适合评测脚本和简单调用，流式接口适合前端逐步展示模型生成内容，提升用户体验。

接口层实现了文本评估、文件上传评估、知识库状态查询和知识库重建等能力。对于文件上传接口，API 会先调用预处理模块提取文本，再把标准化后的内容交给 Agent。响应中保留了原始来源摘要、预处理文本和预处理提示，方便用户知道系统实际读取了哪些资料。

流式接口的设计是本周的重点。系统会依次向前端发送 `rewrite`、`route`、`knowledge`、`specialist_result`、`token` 和 `done` 等事件。这样前端不仅能看到最终回答，还能看到系统如何改写输入、选择哪些专科、检索了哪些知识依据，以及最终回答的生成过程。

本周还实现了根路径服务状态和 Swagger 文档，方便调试接口。通过 API 封装，项目从本地脚本进入可服务化阶段。下一周将继续集成用户系统、会话持久化和上下文记忆，使系统更接近真实应用。

## 第10周：实现用户认证、会话管理与上下文记忆

本周主要完成系统持久化能力。首先在 `database.py` 中实现 MySQL 连接与表结构初始化，包含 `users`、`conversations` 和 `messages` 三张表。服务启动时会自动创建数据库和表，减少部署时的手动配置。用户表保存账号和密码哈希，会话表保存对话标题和模式，消息表保存用户与助手的历史消息及元数据。

认证模块 `auth.py` 使用 PBKDF2 对密码加盐哈希，避免明文保存密码；登录后系统生成 HMAC 签名的 Bearer token，并设置有效期。API 层增加注册、登录、获取当前用户等接口，后续会话接口都通过 Authorization 请求头识别用户。

本周还实现了 `chat_service.py`，负责新建会话、获取会话列表、读取消息历史、更新标题、删除会话和插入消息。为了让系统支持连续健康评估，我实现了轻量上下文记忆：在用户发起新一轮评估时，系统会把最近若干条历史消息拼入当前输入，提示 Agent 只将其作为背景，不虚构新的病史或检查结果。

本周遇到的重点问题是上下文不能无限增长，否则会增加 Token 成本并引入噪声。因此系统限制最多读取 10 条历史消息，并对单条历史内容做长度截断。完成本周工作后，系统已经从一次性问答升级为可登录、可保存、可连续追踪的健康评估应用。

## 第11周：开发 Vue 前端健康评估工作台

本周开始开发前端页面，目录为 `frontend/`，技术栈采用 Vue 3 和 Vite。前端目标不是做展示型网页，而是做一个可实际使用的健康评估工作台。页面包含登录/注册入口、会话列表、新建健康评估、模式切换、消息区、输入框、病例文件上传和右侧知识依据面板。

在交互设计上，用户登录后可以创建多个会话，每个会话支持“专科路由”和“全科综合”两种模式。消息发送后，前端通过 SSE 读取后端流式事件，逐步更新评估状态和回答内容。系统会展示路由到的专科、Token 数、耗时，以及知识库检索片段，增强结果可解释性。

文件上传方面，前端支持 PDF 和常见图片格式，用户可以输入文字说明并同时上传体检报告或病例截图。前端将文件和文本一起提交到后端的流式文件接口，由后端完成解析和 Agent 分析。

本周还完善了基础样式，使界面分为左侧会话栏、中间对话区和右侧洞察面板。这样用户既可以像聊天一样提问，也可以看到系统背后的路由和知识依据。下一周将重点进行功能测试、评测数据集构建和系统性能验证。

## 第12周：构建测试集并开展自动化评测

本周进入系统测试与实验验证阶段。为了评价 Agent 是否真正优于直接调用模型，我构建了 `test_data.json`，共包含 80 条健康咨询样本，覆盖血压、血糖、血脂、饮食与体重、睡眠与运动、吸烟、心理健康和社会支持等主题。每条样本包含用户输入、期望路由专科、相关知识文件、参考回答要点、安全要求、难度和风险等级。

随后实现 `evaluate_test_data.py`，用于批量运行测试集。该脚本支持三种 runner：直接调用本地 Agent、调用已启动的 FastAPI 服务，以及直接调用 DeepSeek 作为基线方法。评测结果会自动保存到 `eval_results/` 下，包括每条样本的路由结果、检索文件、回答内容、耗时、Token 消耗和自动指标。

本周重点统计了三类过程指标：专科路由准确率、知识库检索准确率和高风险安全提示覆盖情况。路由评价通过比较系统预测专科和人工标注专科计算 precision、recall、F1 和完全匹配率；知识检索评价通过比较检索到的知识文件和人工标注相关文件计算文件级指标；安全评价则检查高风险样本是否出现就医、急诊、尽快处理等提示，同时排查延误就医倾向。

通过本周实验，系统验证方式从主观试用变成了可复现的批量评测。下一周将进一步引入 LLM-as-a-judge，对 Agent 输出和 DeepSeek 直接输出进行成对比较。

## 第13周：完成成对评审与实验结果分析

本周主要完成实验对比和论文实验章节素材整理。为了评价最终回答质量，我实现了 `judge_pairwise_comparison.py`，使用 DeepSeek 作为评审模型，对 Healthcare Agent 输出和直接调用 DeepSeek 的 baseline 输出进行成对比较。评审时向模型提供用户输入、参考要点、安全要求和两个候选回答，并要求从正确性、覆盖度、依据充分性和安全性四个维度分别评分，最后判断 Agent 更优、DeepSeek 更优或二者相当。

实验结果显示，专科路由 F1 值达到 0.9058，召回率达到 0.9875，说明系统能够较好识别用户问题涉及的健康方向，尤其能够避免漏掉关键专科。知识库检索文件级召回率为 0.8152，说明 RAG 能够检索到大多数相关知识依据，但准确率仍有提升空间。高风险样本中，系统紧急提示覆盖率达到 100%，未检测到延误就医倾向。

在 79 条有效成对评审样本中，Agent 胜出 28 条，DeepSeek 胜出 22 条，二者相当 29 条。若将 Agent 胜出和打平合并，Agent 在 72.15% 的样本中不劣于直接调用模型。尤其在困难样本中，Agent 胜出 7 条，DeepSeek 无胜出，说明多 Agent + RAG 架构在复杂复合型健康咨询中更有优势。

本周也整理了 `eval_results/EXPERIMENT_ANALYSIS.md`，从实验目的、测试集、评价方法、路由效果、检索效果、安全性和最终回答质量等方面形成完整分析。下一周将对论文、文档和验收材料进行最后整理。

## 第14周：完善论文文档并进行程序验收准备

本周主要进行毕业设计收尾工作。首先根据系统实现和实验结果，补充论文中系统设计、核心模块、实验设置和结果分析等内容。系统设计部分重点说明了 Healthcare Agent 的整体架构，包括输入预处理、问题改写、专科路由、知识库检索、多专科并行分析、总结 Agent、FastAPI 服务、MySQL 会话持久化和 Vue 前端工作台。实验部分则引用批量评测结果，说明系统在路由召回、知识检索、安全提示和复杂问题覆盖度方面的表现。

其次，对项目文档进行了统一整理。README 中记录了项目结构、当前能力、安装方式、知识库配置、API 路径和评测命令；`docs/DEVELOPMENT_ALIGNMENT.md` 记录了后端、Agent 和前端之间的接口契约，方便答辩或验收时解释系统流程；评测结果和分析材料保存在 `eval_results/` 目录中，作为论文实验数据依据。

本周还对程序验收流程进行了准备：检查后端能否正常启动，知识库索引能否构建，前端能否登录、创建会话、发送文本和上传文件，专科路由与全科综合模式是否都能正常返回结果，右侧知识依据面板是否能够显示检索来源。对于系统限制也做了整理，例如 OCR 依赖本机 Tesseract，健康建议不能替代医生诊断，RAG 检索仍存在弱相关片段进入上下文的问题。

经过 14 周开发，项目已经形成较完整的毕业设计成果：既有可运行的智能健康评估系统，也有知识库、评测数据、实验脚本、分析报告和论文支撑材料。后续答辩阶段将重点展示系统实际运行流程、Agent 架构创新点、实验结果以及医疗安全边界。
